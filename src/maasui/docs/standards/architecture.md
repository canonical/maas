# Architecture

## TL;DR

- Feature code lives in domain folders under `src/app/<domain>/` — one folder per feature area (e.g. `machines/`, `pools/`).
- Components follow a four-level elevation path: view-specific → domain-shared → app-shared (`base/`) → library.
- Never import a component from one domain into another — promote it to `base/components/` first.
- Every view must call `useWindowTitle` and render `<PageContent>` as its root.
- `src/app/apiclient/` is auto-generated — never edit it by hand.
- All new REST work goes in `src/app/api/query/` using TanStack Query.
- `src/app/store/` is legacy Redux — do not add new slices, actions, or sagas.
- New global hooks go in the appropriate file under `src/app/base/hooks/` and must be exported from `base/hooks/index.ts`.
- TypeScript is strict — do not use `any`; use `TSFixMe` only as a last resort.
- Write function components only — no class components.

## Directory Structure

```
src/
├── app/
│   ├── base/                        # Cross-cutting shared elements
│   │   ├── components/              # Components reusable across 2+ domains
│   │   ├── hooks/                   # Hooks reusable across 2+ domains
│   ├── api/
│   │   └── query/                   # Manually written TanStack Query hooks
│   ├── apiclient/                   # Auto-generated OpenAPI SDK — do not edit
│   ├── store/                       # Redux slices — legacy, do not add to
│   ├── <domain>/                    # e.g. machines/, pools/, zones/
│   │   ├── components/              # Domain-shared components
│   │   ├── hooks/                   # Domain-shared hooks
│   │   ├── views/
│   │   │   └── <ViewName>/
│   │   │       ├── components/      # View-specific components
│   │   │       ├── hooks/           # View-specific hooks
│   │   │       ├── <ViewName>.tsx
│   │   │       ├── <ViewName>.test.tsx
│   │   │       └── _index.scss
│   │   └── urls.ts
├── scss/
│   └── index.scss                   # Global styles entry point
```

### Layer rules

**`src/app/base/`** — anything used across two or more unrelated domains. Components here must not import from any specific domain folder.

**`src/app/api/query/`** — manually written TanStack Query hooks for REST endpoints. All new API integration work starts here.

**`src/app/apiclient/`** — generated by `yarn generate-api-client` from the OpenAPI spec. Never edit files in this directory; re-run generation when the spec changes.

**`src/app/store/`** — legacy Redux Toolkit slices and Redux-Saga middleware. Handles WebSocket messaging and a small number of legacy REST flows. Do not add new slices or sagas.

**`src/app/<domain>/`** — self-contained feature area. Examples from the codebase: `machines/`, `pools/`, `zones/`, `kvm/`, `networks/`, `tags/`, `controllers/`, `devices/`. Only user-navigable pages sit at the root of `views/` — sub-pages nest inside the parent view folder.

## Component Elevation Path

A component starts at the narrowest scope that fits and moves up only when a second consumer needs it.

### Level 1 — View-specific

**Path:** `src/app/<domain>/views/<ViewName>/components/`

Used by exactly one view. Nothing outside that view directory may import it.

### Level 2 — Domain-shared

**Path:** `src/app/<domain>/components/`

Used by two or more views within the same domain. Still inaccessible to other domains.

### Level 3 — App-shared

**Path:** `src/app/base/components/`

Used across two or more domains. `PageContent` and `SectionHeader` are examples of components that live here.

### Level 4 — Library

**Package:** `@canonical/maas-react-components` or `@canonical/react-components`

Framework-generic or cross-project components. Before building a new shared component, check whether one already exists in `@canonical/maas-react-components`.

### Elevation rules

- Never import a view-specific component from outside its own view directory.
- When a second view needs the same component, move it up immediately — do not duplicate it.
- Before building a new Level 3 component, check Level 4 first.

## Standard View Layout

Every view follows the same shell:

```tsx
import type { ReactElement } from "react";

import PageContent from "@/app/base/components/PageContent";
import { useWindowTitle } from "@/app/base/hooks";
import { PoolsListHeader, PoolsTable } from "@/app/pools/components";

const PoolsList = (): ReactElement => {
  useWindowTitle("Pools");

  return (
    <PageContent header={<PoolsListHeader />}>
      <PoolsTable />
    </PageContent>
  );
};

export default PoolsList;
```

Source: `src/app/pools/views/PoolsList.tsx`

### `PageContent`

The root layout container for every view. It renders the secondary navigation sidebar (for Settings and Preferences pages), the main content area, and `<SidePanel />`. Always use it as the outermost element of a view — never build a custom layout shell.

Props relevant to views:

| Prop | Type | Purpose |
|---|---|---|
| `header` | `ReactNode` | Renders the `SectionHeader` above the content area |
| `children` | `ReactNode` | Main page content |

### `SectionHeader`

Used as the `header` prop of `PageContent`. It renders the page title, subtitle, action buttons, tab navigation, and action menu groups via `MainToolbar` from `@canonical/maas-react-components`.

Key props:

| Prop | Type | Purpose |
|---|---|---|
| `title` | `ReactNode` | Page heading |
| `subtitle` | `ReactNode` | Secondary text below the title |
| `loading` | `boolean` | Replaces title with a spinner when true |
| `subtitleLoading` | `boolean` | Replaces subtitle with a spinner when true |
| `buttons` | `ReactElement[]` | Action buttons rendered in the toolbar |
| `tabLinks` | `DataTestElement<TabLink>[]` | Tab navigation links |
| `actionMenuGroup` | `ReactElement` | Action menu rendered below the toolbar |
| `titleElement` | `keyof JSX.IntrinsicElements` | HTML element for the title (default: `"h1"`) |

### `useWindowTitle`

```tsx
useWindowTitle("Pools");
```

Sets `document.title` to `"<title> | <maasName> MAAS"`. Call it once at the top of every view component, before the return statement.

## Custom Hooks Conventions

### `src/app/base/hooks/` file map

| File | Exported hooks |
|---|---|
| `base.ts` | `useWindowTitle`, `useCycled`, `useProcessing`, `useScrollOnRender`, `useScrollToTop`, `usePreviousPersistent` |
| `forms.ts` | `useFormikErrors`, `useFormikFormDisabled` |
| `tables.ts` | `useTableSort` |
| `node.ts` | `useCanEdit`, `useMachineActions`, `useIsRackControllerConnected` |
| `node-networking.ts` | `useIsAllNetworkingDisabled` |
| `dataFetching.ts` | `useFetchActions` |
| `urls.ts` | `useGetURLId` |
| `analytics.ts` | `useGoogleAnalytics`, `useSendAnalytics`, `useSendAnalyticsWhen`, `useUsabilla` |
| `intro.ts` | `useCompletedIntro`, `useCompletedUserIntro` |

Complex hooks that maintain their own state, effects, or tests live in a dedicated subfolder. Current examples: `usePagination/`, `useDebouncedValue/`, `useDarkMode/`, `useSessionExtender/`.

### When to extract to a hook

Extract logic into a hook when any of the following is true:
- The logic spans three or more statements.
- It involves `useEffect` or `useRef`.
- It is shared across two or more components.

### Placement rules

- Domain-specific hooks belong in `<domain>/hooks/`, not `base/hooks/`.
- New global hooks go in the appropriate existing file under `base/hooks/` (or a new subfolder if the hook is complex).
- Every hook in `base/hooks/` must be exported from `src/app/base/hooks/index.ts`.

### Do

```ts
import { useWindowTitle } from "@/app/base/hooks";
```

### Don't

```ts
import { useWindowTitle } from "@/app/base/hooks/base";
```

Always import from the barrel `index.ts`, not directly from the implementing file.

## API Layer Split

### `src/app/apiclient/`

Auto-generated from the MAAS OpenAPI specification via `yarn generate-api-client`. Contains typed request functions and response types. Never edit files here — changes will be overwritten on the next generation run. Import types from `@/app/apiclient`.

### `src/app/api/query/`

Manually written TanStack Query hooks. All new REST integration work goes here. Each file maps to a domain (e.g. `pools.ts`, `zones.ts`). Hooks wrap `useWebsocketAwareQuery` and `queryOptionsWithHeaders`.

### `src/app/store/`

Legacy Redux Toolkit slices. Use only to read WebSocket-driven state that has not yet been migrated to TanStack Query. Do not add new slices, actions, or sagas. The migration path is: write TanStack Query hooks in `api/query/`, add MSW resolvers in `src/testing/resolvers/`, replace `useSelector` + `useFetchActions` consumers, then remove the slice.

## Dos and Don'ts

**Do** place new views under `src/app/<domain>/views/<ViewName>/`.

**Don't** place a view anywhere else in the tree — only user-navigable pages sit at the root of `views/`.

---

**Do** use generated types from `@/app/apiclient`.

**Don't** edit `src/app/apiclient/` — it is auto-generated.

---

**Do** call `useWindowTitle` in every view.

**Don't** set `document.title` directly.

---

**Do** wrap every view in `<PageContent>`.

**Don't** build custom layout shells per-view.

---

**Do** put new REST hooks in `src/app/api/query/`.

**Don't** add new REST endpoints to `src/app/store/` — Redux handles WebSocket messaging only.

---

**Do** import cross-domain shared components from `base/components/`.

**Don't** import a component from `src/app/<domainA>/` into `src/app/<domainB>/` — promote it to `base/components/` first.

---

**Do** export new global hooks from `src/app/base/hooks/index.ts`.

**Don't** add a hook to `base/hooks/` without adding it to the barrel export.

---

**Do** use `TSFixMe` (importable from `@/app/base/types`) as a last resort with a comment explaining why.

**Don't** use `any`.

---

**Do** write function components.

**Don't** write class components.
